여러분, 혹시 최근 클라우드 환경에서 발생한 대규모 서비스 해킹 소식 들으셨나요? 대부분의 공격은 개발 초기 단계나 배포 과정, 즉 CI/CD 파이프라인과 Kubernetes 클러스터의 취약점을 노립니다. 이렇게 빠르게 변화하는 클라우드 환경에서 DevSecOps 엔지니어들은 보안과 효율성 사이에서 끊임없이 고민할 수밖에 없습니다. 오늘은 GitHub Actions 보안, Kubernetes RBAC, Pod Security Standards, Network Policy, 그리고 Kyverno, Falco, Cosign 같은 도구를 활용한 이미지 및 런타임 보안 전략까지, GitOps 환경에서의 DevSecOps 실전 가이드를 총망라해드리겠습니다. 이 강의를 통해 여러분은 개발 초기 단계부터 런타임까지, 클라우드 네이티브 애플리케이션의 보안을 완벽하게 책임지는 실질적인 노하우를 갖추게 되실 겁니다.

자, 그럼 클라우드 보안 전문가의 길을 함께 시작해볼까요?

가장 먼저 살펴볼 부분은 CI/CD 파이프라인 보안 기초입니다. 왜 CI/CD 보안이 중요할까요? 파이프라인은 코드를 빌드하고 배포하는 과정의 핵심이기 때문에, 이곳이 뚫리면 모든 시스템이 위험해집니다. 개발 초기 단계부터 보안을 적용하는 'Shift-Left Security'의 최전선이라고 할 수 있죠.

GitHub Actions를 예로 들어볼게요. 여러분의 CI/CD 파이프라인이 GitHub Actions를 사용한다면, 가장 먼저 permissions를 최소화해야 합니다. 예를 들어, 빌드 과정에서 쓰기 권한이 필요 없다면 읽기 전용으로 설정하는 것이 기본입니다. 또한, API 키나 데이터베이스 비밀번호 같은 Secret은 절대 코드에 하드코딩하지 말고, GitHub Secrets나 Vault 같은 전용 도구를 사용해 안전하게 관리해야 합니다.

그리고 파이프라인에 보안 테스트 도구를 통합하는 것이 필수입니다.
SAST(정적 애플리케이션 보안 테스트) 도구로는 Semgrep이나 SonarQube를 사용해서 코드의 잠재적 취약점을 미리 찾아낼 수 있습니다. Secret 스캐닝 도구인 Gitleaks나 TruffleHog를 활용하면 실수로 커밋된 민감 정보를 놓치지 않을 수 있죠. Trivy 같은 SCA(오픈소스 의존성 취약점 스캔) 도구는 여러분이 사용하는 라이브러리의 알려진 취약점을 분석해줍니다. 마지막으로 DAST(동적 애플리케이션 보안 테스트)는 OWASP ZAP을 통해 실제로 동작하는 애플리케이션의 취약점을 테스트할 수 있습니다. 이 모든 것을 CI/CD 파이프라인에 자동화하면, 보안 결함을 빠르게 발견하고 수정할 수 있습니다.

다음은 Kubernetes RBAC 보안입니다. Kubernetes 환경에서 누가 무엇을 할 수 있는지를 정의하는 RBAC는 최소 권한 원칙을 적용하는 핵심 요소입니다. RBAC는 크게 Role과 ClusterRole, 그리고 RoleBinding과 ClusterRoleBinding으로 구성됩니다. Role은 특정 네임스페이스 내에서의 권한을, ClusterRole은 클러스터 전체의 권한을 정의하죠. 예를 들어, 개발팀에게는 특정 네임스페이스에서 Pod를 생성하고 로그를 볼 수 있는 Role을 부여하고, 운영팀에게는 클러스터 전체 리소스를 관리할 수 있는 ClusterRole을 부여하는 식입니다. ServiceAccount 역시 중요한데, Pod가 Kubernetes API와 통신할 때 사용하는 계정이기 때문에, 이 ServiceAccount에도 필요한 최소한의 권한만 부여해야 합니다. automountServiceAccountToken: false 설정을 통해 불필요한 토큰 마운트를 막는 것도 좋은 방법입니다.

세 번째는 Pod Security Standards (PSS)입니다. PSS는 Pod가 가질 수 있는 보안 속성의 집합을 정의하는 Kubernetes의 기본 보안 기능입니다. Privileged, Baseline, Restricted 세 가지 프로파일이 있는데, 일반적으로 Restricted 프로파일을 적용하여 가장 엄격한 보안을 유지하는 것이 좋습니다. Namespace 레벨에서 PSS를 강제하면 해당 Namespace 내의 모든 Pod에 일관된 보안 정책을 적용할 수 있습니다. 예를 들어, securityContext를 통해 컨테이너가 root 권한으로 실행되지 않도록 runAsNonRoot: true를 설정하거나, 파일 시스템 권한을 제한하는 readOnlyRootFilesystem: true를 사용하는 것이 모범 사례입니다. allowPrivilegeEscalation: false 설정은 컨테이너 내에서 권한 상승 공격을 방지하는 데 매우 효과적입니다.

네 번째는 Network Policy 구현입니다. Kubernetes 클러스터 내부의 Pod 간 통신을 제어하는 Network Policy는 네트워크 격리의 핵심입니다. 초기에는 모든 트래픽을 허용하는 대신, default-deny 정책을 먼저 적용하고 필요한 트래픽만 명시적으로 허용하는 것이 가장 안전한 접근 방식입니다. 예를 들어, 웹 서비스 Pod는 데이터베이스 Pod와 80번 포트로만 통신할 수 있도록 제한하고, 다른 Pod로부터의 인바운드 트래픽은 모두 차단하는 것이죠. 이를 통해 측면 이동(Lateral Movement) 공격의 가능성을 크게 줄일 수 있습니다.

다섯 번째는 Secret 관리입니다. Kubernetes Secret은 민감 정보를 저장하는 데 사용되지만, 기본적으로 Base64 인코딩만 되어 있어 암호화되어 있지 않습니다. 따라서 더 안전한 Secret 관리가 필요합니다. External Secrets Operator는 AWS Secrets Manager, Azure Key Vault, GCP Secret Manager 같은 외부 Secret 스토어와 Kubernetes Secret을 동기화하여 관리 부담을 줄이고 보안을 강화합니다. GitOps 환경에서는 Sealed Secrets가 유용합니다. 민감 정보를 암호화하여 Git 저장소에 안전하게 저장하고, Kubernetes 클러스터에서만 복호화하여 사용할 수 있게 해주죠. HashiCorp Vault와 같은 중앙 집중식 Secret 관리 솔루션을 통합하는 것도 강력한 선택입니다.

여섯 번째는 이미지 보안입니다. 여러분이 사용하는 컨테이너 이미지가 안전한지 검증하는 것은 매우 중요합니다. Admission Controller는 Pod가 클러스터에 배포되기 전에 특정 정책을 만족하는지 검사하는 Kubernetes의 핵심 기능입니다. Kyverno 같은 정책 엔진을 활용하면, 서명되지 않은 이미지를 차단하거나, 특정 레지스트리에서 온 이미지만 허용하는 정책을 쉽게 구현할 수 있습니다. 예를 들어, Cosign으로 서명된 이미지만 클러스터에 배포되도록 강제하여 이미지 무결성을 보장할 수 있습니다. 또한, Trivy 같은 취약점 스캐너와 연동하여 알려진 취약점이 있는 이미지는 아예 배포되지 않도록 차단하는 정책을 설정할 수도 있습니다.

일곱 번째는 런타임 보안입니다. 배포된 Pod가 실행 중일 때 발생하는 의심스러운 활동을 탐지하는 것은 보안의 마지막 방어선입니다. Falco는 시스템 호출 레벨에서 비정상적인 행위를 탐지하는 런타임 보안 도구입니다. 예를 들어, 컨테이너 내에서 예상치 못한 셸 실행, 민감한 파일 접근, 네트워크 연결 시도 등을 실시간으로 감지하고 경고를 발생시킬 수 있습니다. 미리 정의된 규칙 세트를 활용하거나, 여러분의 환경에 맞춰 사용자 정의 규칙을 추가하여 보안 가시성을 극대화할 수 있습니다.

이제 2025년 Kubernetes 보안 업데이트의 주요 내용을 살펴보겠습니다. Kubernetes는 2024년 말 1.32 "Penelope"부터 2025년 12월 1.35 "Timbernetes"까지 보안 기능을 대폭 강화했습니다.

첫째, Fine-grained Kubelet API Authorization (KEP-2862)입니다. Kubelet API에 대한 접근 제어가 더욱 세밀해져, 노드별, Pod별로 Kubelet API 접근 권한을 정교하게 제어할 수 있게 되었습니다. 이는 측면 이동 공격을 방지하고 침해 발생 시 피해 범위를 최소화하는 데 큰 도움이 됩니다.

둘째, Credential Tracking for Forensics입니다. 인증서 서명 기반의 credential ID 생성을 통해 포렌식 기능이 강화되었습니다. 어떤 사용자가 어떤 Credential을 사용했는지 추적하기가 훨씬 쉬워져, 침해 사고 발생 시 원인 분석 및 대응 속도를 높일 수 있습니다.
